## Methodology

**Phase 1 — filtered smoke.** Ran the cache fetch with `limit=50` against the live `deftai/directive` issue stream. Goal: validate scanner v2.0.0 end-to-end on a known-bounded slice and produce a clean baseline before scaling.

**Phase 2 — unfiltered smoke (attempted).** Ran `task triage:bootstrap` (which internally calls `cache:fetch-all`) against the full backlog. `cache:fetch-all` itself completed (~322 issues by ~18:00 UTC); the surrounding orchestrator never returned. After 71+ minutes of audit-log silence the maintainer cancelled the run. This salvage report is built from the cache + audit jsonl that survived on disk; no further network fetches were performed.

Read-only verification only:
- `Get-ChildItem .deft-cache\github-issue -Recurse -Directory` to count cached issue dirs.
- `Get-ChildItem .deft-cache -Recurse -File | Measure-Object -Property Length -Sum` for total bytes.
- `Get-Content .deft-cache/quarantine-audit.jsonl` for line count, last entry, and JSON parse via Python.

## Findings

### Cache + scanner layer (v0.26.0)

- **Scanner v2.0.0 holds at backlog scale.** 319 issues, 0 hard-fails, 0 malformed audit rows, uniform scanner version, 100% `scan_passed` / `content_written`. No regression observed vs. Phase 1.
- **Audit jsonl format is stable.** Both phases parse cleanly via stdlib `json`.
- **TTL + meta schema confirmed on Phase 1 spot-check.** Not re-checked in Phase 2 (read-only salvage), but no audit-side anomalies suggest meta drift.

### Detector tuning evidence — feeds [#949](https://github.com/deftai/directive/issues/949)

Both phases produced an **80–85% `injection-heading / fence-and-pass`** rate against organic issue bodies. That is a high benign-flag floor against a normal backlog. **Recommend revisiting the heading detector's specificity (e.g. require an injection-shaped heading rather than any heading) before adding new detector categories.** Phase 1 metrics have been posted as an evidence comment on #949.

### Defect A — task description references obsolete `--skip-gitcrawl`

`task triage:bootstrap` task description still references a `--skip-gitcrawl` flag that `scripts/triage_bootstrap.py` does not accept (Story 3 leftover). Cosmetic / docs-side; no runtime impact, but breaks any user who copy-pastes the documented flag. Filed as [#951](https://github.com/deftai/directive/issues/951). **Not fixed in this PR.**

### Defect B — bootstrap orchestrator hangs after `cache:fetch-all` at backlog scale (HIGH)

`task triage:bootstrap` ran `cache:fetch-all` to apparent completion (~322 issues by 18:00 UTC) and then the orchestrator never returned control. Audit log went silent at `2026-05-06T18:00:09Z`; maintainer cancelled after 71+ min. This is a real Story 3 regression that only manifests at backlog scale; the Phase 1 filtered smoke (limit=50) did not exercise it. Filed as [#952](https://github.com/deftai/directive/issues/952). **HIGH priority — blocks bulk-ingest stage 2 and the #946–#949 follow-up work.** **Not fixed in this PR.**

### Defect C — meta finding (agent runtime, not deft)

The agent driving the smoke became unresponsive to its inbox while blocked on a long-running tool call (the hung `task triage:bootstrap`), forcing the maintainer to cancel from the UI rather than via an inbox signal. This is **out of scope for `deftai/directive`** and is recorded here as a meta finding only. **Not filed as a directive issue.**
