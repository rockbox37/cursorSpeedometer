
Waited ~10 minutes for the per-hour reset, then retried successfully. The rate-limit hit is incidental to this smoke -- it's the listing step (`gh issue list --json`) that's metered, not the per-key cache work.

## Findings

### #952 / #955 -- bootstrap watchdog and structured progress (RESOLVED)

The orchestrator hang surfaced in [#952](https://github.com/deftai/directive/issues/952) is **fixed by [#955](https://github.com/deftai/directive/pull/955)**. Direct evidence:

- Bootstrap completed cleanly on the first attempt at full backlog scale (310 + 10 = 320 issues), exit `0`, wall-clock 448.6 s. The prior run hung at the same step after 71+ min of silence.
- The 3600 s watchdog was configured but **did not fire** -- `populate_cache` finished in 448.265 s, well under the cap. Watchdog availability is the safety net; the actual fix is whatever PR #955 changed in the orchestrator's exit path (subprocess plumbing).
- Per-step structured progress (`triage:bootstrap step <i>/<N> ... starting/done (...)`) was emitted on stderr for all 4 steps with timestamps the operator can correlate. `--json` mode emitted one structured object per step on stdout. Both observability surfaces are usable for diagnosing future regressions without re-cracking the orchestrator.

**No regression observed; no follow-up needed.** The watchdog observability worked as designed -- it just didn't have to.

### #949 -- detector tuning (full-scale evidence)

At full backlog scale (N=320):
- **271 / 320 (84.7%)** rows flagged `injection-heading`, every one at severity `fence-and-pass`.
- **2 / 320 (0.6%)** rows flagged `invisible-unicode` at severity `strip-and-pass`.
- **48 / 320 (15.0%)** rows clean.

The injection-heading rate is now confirmed across three independent samples (Phase 1 N=50, prior Phase 2 N=319, this run N=320) within ~5 percentage points. **An 85% benign-flag rate against an organic backlog is high.** The current detector flags any markdown `## Heading` it can't prove benign. Recommended #949 follow-up direction: tighten the heading rule to require an injection-shaped heading (e.g. `## Ignore previous instructions`, `## SYSTEM PROMPT`, embedded role tokens) before fence-wrapping, rather than wrapping every heading. This is a precision fix; the current `fence-and-pass` severity ensures false positives still pass through to consumers, so this is a UX / signal-to-noise concern rather than a correctness regression.

`invisible-unicode` at 2 / 320 is a sensible floor and not a tuning candidate.

### #947 -- cache size sizing

Empirical sizing for v0.26.0 caches at this codebase's scale:
- **Per issue:** ~9.7 KB on disk (`3,173,052 B / 320 issues`). Three files per issue (`meta.json`, `content.md`, `raw.json`).
- **Linear scale:** for a 1000-issue backlog expect ~9.7 MB; for 5000 issues expect ~48 MB.
- **Audit jsonl bytes per row:** ~339 B (`108,505 B / 320 lines`). Audit growth is the bottleneck only at 100k+ events; for the v0.26.0 release it's not a sizing concern.

Sizing-wise, the cache is small. No infrastructure changes warranted before promotion.

### #948 -- audit-log growth pattern

Phase 2 + Phase 2B together produced **320 audit rows over ~21 minutes** (one per `cache:put` of a previously-uncached or stale entry). Phase 2B's idempotent re-run added **0 audit rows for the 310 skip-fresh entries** -- skip-fresh paths do not write to the audit log, only `cache:put` events do. This is the right design: audit-log growth tracks new content, not re-fetches.

Linear extrapolation: at one full-backlog refresh / week (TTL=7d), the audit log grows by ~320 rows / week / 1000-issue repo, i.e. ~106 KB / week. Bound rotation policy is comfortably 12+ months out for a codebase this size.
