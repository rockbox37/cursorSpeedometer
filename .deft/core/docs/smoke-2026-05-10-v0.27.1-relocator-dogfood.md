## Scope finding -- why state-B-primary plus a synthetic state-A

The 2026-05-10 org-wide scan of `deftai`'s 38 repos found that every consumer of the deft framework today is either canonical-shape (`.deft/core/`) or absent. Pure-legacy state-A installs do not exist in the wild because the broken installer line predates the existence of any legacy `deft/` consumer; pure-hybrid state-C installs do not exist for the same reason. The three documented broken-install state-B consumers are `deftvisage`, `evolution`, and `postmortem`. Hence:

- **Target 1** runs the relocator against a real state-B consumer (`deftai/postmortem`). The active vBRIEF's `Test` narrative for `992-ac-5` calls out "2-3 deftai repos covering states A, B, C"; the real-world finding is that B is the only live broken state, so the rc1 dogfood centres on B.
- **Target 2** synthesises a pure state-A consumer locally so the canonical no-`--force` happy-path is exercised end-to-end. The fixture pins the legacy `deft/` deposit to the **v0.27.1 archive** so the framework hash compare returns "not customised" and the state classifier returns `A` (not `E`). A v0.27.0-pinned fixture was also evaluated and is documented under "Notes" below -- it classifies as `E` because the v0.27.0 byte content differs from the v0.27.1 framework source, which is the realistic upgrade-from-prior-minor shape the `--force` gate is designed for.

The relocator's seven phases (state classify -> preflight -> snapshot -> wipe -> deposit -> AGENTS.md re-render -> .gitignore upkeep -> advisory grep) were exercised end-to-end on both targets. Snapshot-rollback was verified on both targets.

## Methodology

Both targets are throwaway clones / synthetic git repos **outside** the framework worktree at `C:\Repos\deft\directive-992-dogfood`. **No commits or pushes were made to either dogfood subject** -- they were inspected for byte-clean status and then discarded.

The relocator was invoked from the framework repo's `scripts/relocate.py` against each target via `uv run python`. The `--framework-source` flag pointed at the v0.27.1 framework checkout for every invocation; `--project-root` pointed at the target. Dry-run was run first for plan capture; the apply step used `--confirm` (non-interactive); rollback used `--rollback` against the most recent snapshot tarball.

State detection was driven by `scripts/_relocate_states.detect_install_state(project_root, framework_source=...)` so the classifier's pre-flight gates (state E "customised framework dir", state G "active swarm worktree") were given the same `framework_source` they would receive at relocate time. This is the load-bearing wiring fix that distinguishes "broken state-B installer output" (state B per the layout-only classifier) from "broken state-B installer output but framework bytes drifted from current release" (state E under the customisation pre-flight) -- and the rc1 dogfood deliberately exercises both.

## F1 / F2 / F3 verification

The v0.27.1 follow-up trio (#1015) closed three follow-up findings on the v0.27.0 relocator. Each is explicitly checked below:

### F1 -- self-bootstrap from in-place tree (no webinstaller dependency)

The v0.27.0 relocator's `_running_inside_wipe_target` self-detect would fail loud when the relocator was invoked from inside a wipe target (e.g. `<consumer>/.deft/core/scripts/relocate.py` running against the same `<consumer>`). The v0.27.1 fix replaced fail-loud with an in-process self-bootstrap: the framework is copied to an OS temp directory and the relocator is re-launched from the temp copy with a `--bootstrapped-from-temp` sentinel so the in-place tree is never holding live import handles into its own wipe target.

For both dogfood runs the relocator was invoked from the **framework repo's** `scripts/relocate.py` (path: `C:\Repos\deft\directive-992-dogfood\scripts\relocate.py`) against targets under `C:\Temp\dogfood-state-*`. The self-detect therefore returned `False` for every invocation (script does NOT live inside any wipe target) and the bootstrap path was not entered. This is the canonical "no webinstaller dependency" shape: the relocator ran end-to-end from an external framework copy directly against the consumer project. **F1's load-bearing claim -- the v0.27.1 relocator can run without a webinstaller bootstrap when an external framework copy is available -- is verified.** The exhaustive bootstrap-triggered path is covered by `tests/relocate/test_self_bootstrap.py` and is not re-exercised here.

### F2 -- canonical-default gitignore set (`.deft/core/` NOT auto-ignored)

`scripts/relocate.py::GITIGNORE_LINES = (".deft-cache/", "vbrief/.eval/")` -- the v0.27.1 canonical default deliberately excludes `.deft/core/` because per #11 the framework deposit ships with the consumer's repo as read-only packaged assets, and auto-gitignoring it would silently break that contract on every v0.27.0 install already in the wild.

Verified on both targets post-relocate: `.gitignore` contains exactly the `.deft-cache/` and `vbrief/.eval/` lines added under the "Added by deft relocator (#992 PR2)" comment marker; **zero `.deft/core/` entries**. **F2 verified.**

### F3 -- rollback-tracked-paths byte-clean contract

`scripts/_relocate_snapshot.py::ROLLBACK_TRACKED_PATHS = (LEGACY_FRAMEWORK_DIR, CANONICAL_FRAMEWORK_DIR, "AGENTS.md", ".gitignore")`. The v0.27.0 relocator's rollback path restored captured paths but left relocator-created post-snapshot residue in place; specifically, a relocator-created `.gitignore` would persist in `git status --porcelain` even when the pre-relocate project had no `.gitignore`. The v0.27.1 fix clears every tracked path before extracting the tarball so uncaptured paths (residue) stay absent.

Verified on both targets post-rollback: `git status --porcelain` returns `?? .deft-cache/` only -- which is the snapshot tarball directory itself, **intentionally excluded** from `ROLLBACK_TRACKED_PATHS` so re-rollback against the same snapshot remains possible. On Target 2 the relocator-created `.gitignore` (the pre-relocate fixture had none) was correctly removed during rollback; on Target 1 the pre-existing `.gitignore` was restored byte-identical (SHA-256 match against pre-relocate). **F3 verified.**

## Self-bootstrap verification (no webinstaller dependency observed)

Both targets ran the v0.27.1 relocator end-to-end **without any webinstaller invocation**. The relocator was invoked directly via `uv run python scripts/relocate.py ...` from the framework worktree; the `--framework-source` flag pointed at the same worktree (i.e. an external-to-the-wipe-target framework copy). This is the F1 follow-up's intended consumer UX: an operator with the deft framework checked out anywhere on disk can relocate any consumer project without first fetching `upgrade.sh` / `upgrade.ps1` from `<webinstaller>/`. The bootstrap fetches-fresh-to-temp shape remains supported for the in-place invocation case (a consumer running `<consumer>/.deft/core/scripts/relocate.py` against `<consumer>` itself) -- that path is covered by the in-tree `test_self_bootstrap` suite.

## Notes

### Why two synthetic-state-A passes were considered

An initial synthetic state-A fixture pinned the legacy `deft/` deposit to the **v0.27.0** archive (one release behind v0.27.1). That fixture classified as state-**E** under the v0.27.1 relocator because the v0.27.0 framework bytes differ from the v0.27.1 framework source (the `is_framework_customized` check returns True). This is the realistic upgrade-from-prior-minor shape and is what the `--force` gate is designed for; it is not a defect.

The fixture documented in "Target 2" above pins the deposit to **v0.27.1** so the framework hash compare returns "not customised" and the state classifier returns `A`. That exercises the canonical no-`--force` state-A happy path. The v0.27.0-pinned variant was rebuilt with v0.27.1 to ensure the rc1 dogfood explicitly demonstrates the relocator's smooth state-A handling; the v0.27.0-vs-v0.27.1 drift path is structurally identical to Target 1's real-world state-E shape and gets full coverage there.

### PowerShell 5.1 / non-ASCII glyphs (#798)

The dogfood host runs `pwsh 7.6.1`, exempt from rule #798. This evidence document was written via `create_file` (Python `pathlib` under the hood) for the same UTF-8 safety the rule mandates on PS 5.1, even though the host shell here does not require it. The em dash in postmortem's pre-relocate `AGENTS.md` (rendered as `ΓÇö` in the relocator's advisory console output) is a pwsh console codepage rendering artifact, not a file-content corruption -- the underlying file bytes were never touched by the relocator's read-side decode.
